  Entscheidung: Welle 10 — Security-Welle in vier Schnitten: Container-Image-Vulnerability-Scan in CI (trivy soft-fail), Strict Security-Header in Nginx (HSTS/CSP/X-Frame/Referrer/Permissions/X-Content-Type), Upload-MIME + Size-Limit fuer Admin-Imports, zusaetzlicher Per-Account-Login-Rate-Limit oben drauf zum bestehenden Per-IP-Bucket.
  Begruendung: Der Lerneffekt aus dem Provider-Quellen-Audit war "kein klassisches Anti-Malware noetig solange wir nur JSON akzeptieren — aber CSP/HSTS, Container-Scan, Upload-Validation und Per-User-Login-Rate-Limit sind ueberfaellig". Ohne CSP koennte ein gestolener Plaintext-Cookie XSS-anfaellige Stellen finden; ohne Upload-Size-Limit kann ein Admin-Account-Kompromiss einen Worker mit beliebig grossen Files in den OOM treiben; ohne Per-Account-Rate-Limit umgeht ein IP-rotierender Brute-Force-Angreifer den existierenden IP-Bucket trivial.
  Konsequenzen: (a) Container-Scan: `aquasecurity/trivy-action@master` in `ci.yml` und `publish.yml` als zwei Steps (backend + frontend image); severity HIGH+CRITICAL, ignore-unfixed, exit-code 0 (informativ statt blockierend; spaeter auf hard-fail umstellbar). Neue `.trivyignore` initial leer mit dokumentiertem Format. (b) Security-Header: `frontend.nginx.conf` ergaenzt um `Strict-Transport-Security max-age=31536000; includeSubDomains`, `X-Content-Type-Options nosniff`, `X-Frame-Options DENY`, `Referrer-Policy strict-origin-when-cross-origin`, `Permissions-Policy` (geo/mic/cam/payment/usb alle disabled), und CSP tuned fuer Vite/React (`script-src 'self' 'unsafe-inline'`, `style-src 'self' 'unsafe-inline'` fuer Tailwind, `connect-src 'self' https: wss: ws:` fuer API + Alpaca-Stream, `frame-ancestors 'none'`, `object-src 'none'`). (c) Upload-Validation: neuer Helper `_read_admin_upload_json(file)` in `main.py` checkt Content-Type-Allowlist (`application/json`, `application/octet-stream` als Fallback fuer Browser-File-Picks ohne Endung) UND streamt Bytes inkrementell mit Size-Cap `ADMIN_UPLOAD_MAX_BYTES` (default 50 MB, env-overridable). Beide Admin-Endpunkte (`/api/admin/import`, `/api/admin/backups/import`) nutzen ihn. HTTP 400 bei Type-Mismatch / Invalid-JSON, HTTP 413 bei Size-Cap. (d) Per-Account-Rate-Limit: zweiter Bucket-Scope `login_per_account` (10/15min, gegenueber 5/5min per-IP) keyed auf email; beide Buckets in der Login-Route gleichzeitig enforced. 5 neue Unit-Tests fuer Upload-Validation: JSON akzeptiert, octet-stream akzeptiert, text/html abgelehnt (400), oversized abgelehnt (413), Invalid-JSON abgelehnt (400).   Entscheidung: Phase 3 wird mit dynamischer Slippage (skaliert mit Position-Size relativ zum Tagesvolumen, gecappt bei MAX_SLIPPAGE_PCT=1%) und asset-spezifischen Fee-Multipliers (Stock/ETF 1x, Crypto 5x ueber `FEE_MULTIPLIER_BY_ASSET_CLASS`) abgeschlossen.   Konsequenzen: GitHub Actions wird zum kontinuierlichen Sync-Pfad fuer Integrationsstaende; `latest` ist nur ein beweglicher Integrationszeiger, produktive Upgrades bleiben bei expliziten Versionstags; fuer den Automatikpfad muessen funktionierende Docker-Hub-Secrets im Repository hinterlegt bleiben.
